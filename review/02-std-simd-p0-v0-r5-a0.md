# `std::simd` 审查答复 (p0-v0-r5-a0)

对应报告：`review/02-std-simd-p0-v0-r5.md`

本答复基于当前工作区代码状态（截至 2026-03-20）。`r5` 提供了有价值的补充审查线索，但其中有若干规范归因和总体结论并不成立；因此本轮不按“整份报告原样接受”处理，而按以下三类处置：

- 接受：问题判断成立，按后续轮次直接修复或补测
- 部分接受：现象或补测方向成立，但 `r5` 的规范依据、严重程度或建议修法不完全成立
- 驳斥：`r5` 的问题定性不成立，不按其建议修改实现；如仍有价值，则转化为补充测试或口径澄清项

后续执行顺序固定为三轮：

1. 接受项
2. 部分接受项中可安全落地的部分
3. 驳斥项后的补充完善

## 总体结论

对 `r5` 的总体判断如下：

- 不接受“`r5` 可直接作为最终关闭前结论”的判断
- 不接受“P1928R15 覆盖率约 97%”这一总结性口径
- 不接受 `I-2` 和 `I-1` 在 `r5` 中给出的规范归因
- 接受其作为“补充问题与测试 backlog”的价值

这意味着：

- `r4` 的“可以直接关闭审查循环”结论需要撤回
- 但 `r5` 也不能按“发现了一个已证实的高严重度标准缺陷”来整体采纳

Forge 对 backport 的最高要求仍然是“未来标准库原生支持后，下游零修改切换”；因此本轮优先处理：

- 会影响后续维护判断的死代码和错误说明
- 缺失但低风险的测试覆盖
- 可澄清而不必过度修改实现的行为边界

## 对实现问题的逐条答复

### I-1 `to_ullong()` 宽掩码静默截断

结论：部分接受。

- 我们接受 `r5` 对“当前实现缺少 `>64 lane` 场景覆盖”的观察。
- 也接受：当前 `to_ullong()` 在宽掩码下会忽略第 64 位及以上的 `true` 位，行为边界需要被测试显式锁定。
- 但不接受 `r5` 中“P1928R15 明确要求这里抛 `overflow_error`”的规范归因，也不接受基于该归因得出的“高严重度标准缺陷”定性。

本轮处理方式：

- 第二轮补宽掩码 `to_ullong()` 测试
- 不先入为主地把测试写成“必须抛异常”
- 只修正我们确认成立的部分：行为边界未覆盖、语义需要澄清

### I-2 `operator[]` 返回值而非代理引用

结论：驳斥。

- 不接受 `r5` 中“P1928R15 要求通过 `vec[i] = value` / `mask[i] = true` 修改 lane，因此必须返回代理引用”的结论。
- 当前实现中 `basic_vec::operator[]` 与 `basic_mask::operator[]` 的按值返回行为，已由 [`test/test_simd_api.cpp`](/home/i/projects/forward/ccforge/test/test_simd_api.cpp) 明确记录，并且本轮不会改成代理引用模型。

本轮处理方式：

- 第三轮补更明确的编译期测试，锁定当前按值返回和“下标赋值不是支持入口”的口径
- 不修改公开 API 以迎合 `r5` 的这条建议

### I-3 `reduce_min_index` / `reduce_max_index` 全 false 返回 0

结论：部分接受。

- 我们接受 `r5` 对“当前行为不利于调试”的观察。
- 但这里本来就是前置条件域外场景；因此不接受把它直接上升为明确标准缺陷。

本轮处理方式：

- 第二轮评估是否增加 debug-only 断言
- 若断言会带来不必要的行为扰动，则仅在本答复中将其关闭为 QoI 观察项

### I-4 `detail::cat_impl` 为死代码

结论：接受。

- 该 helper 当前未被调用，属于真实死代码。
- 第一轮删除，降低维护噪音。

### I-5 Include guard 为死代码

结论：接受。

- 我们接受“这不是实际 include guard”的判断。
- 但不会简单删除该宏，因为现有测试使用它观测 backport 头是否被包含。

本轮处理方式：

- 第一轮保留宏定义
- 在实现中补注释，明确其只是 feature/test marker

### I-6 缩进风格不一致

结论：接受，但降级为局部代码质量项。

- 我们接受其属于真实风格问题。
- 但不会为了风格统一做一次全文件纯格式化，以免制造大范围低信号 diff。

本轮处理方式：

- 仅在后续触及的代码块附近顺手统一缩进
- 不单独立项做全文件格式化

## 对测试问题的逐条答复

### T-1 缺少宽掩码 `to_ullong()` 测试

结论：部分接受。

- 接受补测试方向。
- 不接受“修复后应抛异常”这一预设。
- 也不接受“必须依赖 AVX512 平台”这一说法；宽掩码测试可通过固定宽度别名完成。

本轮处理方式：

- 第二轮在 [`test/test_simd_mask.cpp`](/home/i/projects/forward/ccforge/test/test_simd_mask.cpp) 补 `mask<signed char, 128>` 覆盖

### T-2 缺少 `permute` 特殊索引测试

结论：接受。

- 第一轮补 `zero_element` / `uninit_element` 测试。

### T-3 缺少非值保持转换的编译失败负测试

结论：接受。

- 当前实现已通过 `static_assert` 约束无 `flag_convert` 的非值保持转换，但测试未锁定这一点。
- 该项适合转化为负向 compile probe，而不是普通运行时测试。

本轮处理方式：

- 第三轮在 `test/CMakeLists.txt` 中新增负向 probe，并补最小专用源文件

### T-4 缺少 `constexpr` 求值测试

结论：接受。

- 第一轮补 `static_assert` 覆盖，优先验证稳定、无争议的常量表达式路径。

### T-5 类型覆盖有限

结论：部分接受。

- 接受“类型矩阵仍不够系统”的观察。
- 但不接受“现有运行时测试几乎只覆盖 `int` 和 `float`”这一表述；当前已覆盖 `long long` 和 `signed char` mask 等类型。

本轮处理方式：

- 第二轮只做最小高价值扩展，不引入大规模 typed test 重构

### T-6 缺少掩码 + 自定义二元操作的 `reduce` 联合测试

结论：接受。

- 第一轮补 `reduce(value, mask, std::multiplies<>{}, 1)` 运行时测试。
