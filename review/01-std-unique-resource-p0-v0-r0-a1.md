# `std::unique_resource` 审查答复 (v0-r0-a1)
对应报告：`report/01-std-unique-resource-v0-r0.md`
## 先确认本项目的判断原则
在继续讨论具体条目前，先明确本项目对 backport 的核心目标。这些目标会直接决定我们如何看待“问题是否成立”以及“问题优先级如何排序”。
### 原则 1：以未来标准库无感切换为最高目标
本项目不是做一个“功能大致相近”的替代实现，而是要尽可能做到：当未来 Zig/LLVM/libc++（或对应标准库实现）原生提供 `std::unique_resource` 后，下游代码升级工具链时 **无需任何源码修改、无需任何宏分支、无需任何兼容层判断**，即可直接切换到原生实现。
因此，我们追求的不是“能用”，而是：
- API 形状尽可能一致
- 约束与重载参与规则尽可能一致
- 异常规格与失败路径语义尽可能一致
- 通过标准头注入后的用户体验尽可能一致
### 原则 2：额外公开 API 本身就是兼容性风险
只要一个接口在最终标准中不存在，即使它“更方便”或“功能正确”，也会带来兼容性风险。因为下游一旦依赖该接口，将来切换到原生标准库时就会失效。
所以，**非标准扩展不是中性选择，而是向未来不兼容方向偏移**。
这也意味着：
- 额外构造函数不是“可选增强”
- 额外工厂函数不是“语法糖”
- 只要进入正常使用路径，就必须按高优先级处理
### 原则 3：约束形式不是纯风格问题
对于以“无感替换标准库”为目标的 backport，`requires`、`Mandates`、SFINAE、自由函数是否参与重载决议、成员函数是否提供相同约束，这些都不能简单视为“现代化风格问题”。
它们会影响：
- 重载选择结果
- 编译失败位置与诊断形式
- 某些边缘场景是否被接受或拒绝
- 与未来标准库实现的可替换性
因此，只要这些差异可能改变用户代码的编译行为，就应按“标准一致性问题”处理，而不只是代码风格建议。
### 原则 4：测试必须验证标准使用路径
本项目的价值不只是实现了某个类模板，而是要通过 `forge` 的注入机制，让用户像使用未来标准库一样使用它。因此测试不应只验证实现头本身，还应验证：
- 通过 `#include <memory>` 是否可见
- 通过 `forge::forge` 注入后是否成立
- 与标准用法一致的路径是否完整工作
否则，就算实现本体看起来正确，也不能证明“未来无感切换”的目标真的达成。
### 原则 5：异常路径语义一致性应被单独强调
对于资源管理类型，异常路径不是边缘附属行为，而是语义核心的一部分。只要实现或测试在构造失败、赋值失败、移动中途失败等路径上与标准行为不一致，就可能导致：
- 资源泄漏
- 所有权状态不一致
- 下游对异常安全级别形成错误假设
因此，异常路径的一致性不应仅视为原则 1 的子项，而应单独作为优先判断维度。

## 在上述原则下，对本次 review 的总体判断
结论是：**这份 review 总体合理，并且基本符合本项目的核心目标。**
尤其是报告中把下列问题纳入审查范围，这一点是正确的：
- 非标准额外 API
- 约束与标准 wording 的偏离
- 异常安全与失败路径
- 通过标准入口的测试覆盖
也就是说，这份 review 不是“过度挑剔”，而是基本符合本项目应有的严格度。

## 对实现问题的答复
### 应接受并推进修正
- **I-1 `enable_if_t` 而非 `requires`**：接受。这里不能仅按“写法老旧”看待，而应按标准可替换性处理；约束表达方式会影响重载决议与诊断行为，因此优先级不应按普通风格整理处理。
- **I-2 默认构造函数**：接受。该接口若保留，会直接引入标准中不存在的可用写法，破坏未来无感切换。并且它处于正常使用路径，不只是理论偏差。
- **I-3 `make_unique_resource`**：接受。该工厂函数不是最终标准 API，继续暴露会诱导下游形成未来不兼容依赖。若下游广泛采用，它带来的迁移破坏面甚至可能大于个别异常路径缺陷。
- **I-4 移动赋值 `noexcept` 基于 `resource_storage` 而非 `R`**：接受。它更偏向 wording/规格贴合问题，但在本项目目标下仍应修正，以减少与未来标准实现的行为差异。
- **I-5 移动构造异常清理**：接受。这是实质性语义/异常安全问题，应优先修复。
- **I-6 `reset(RR&&)` 使用 SFINAE 而非按标准 Mandates 处理**：接受。这里将报告中的低优先级上调，是因为它不只是“实现写法不同”，而是会改变不满足条件时的编译行为；对于以标准可替换性为目标的 backport，这属于兼容性问题。
- **I-7 `swap` 约束缺失**：接受。是否参与重载决议、何时报错，都是标准接口行为的一部分。
- **I-8 异常处理中的 `is_invocable_v` 检查**：接受。按项目目标，应尽量贴近标准失败路径语义，而不是在实现中加入额外宽松分支。
- **I-9 注入 `std` 命名空间**：接受其“需要文档明确说明”的结论。这里是 backport 机制的工程权衡，应被承认并清楚记录。

## 对测试问题的答复
### 应接受并补齐
- **T-1 `release()` 后析构不调用 deleter**：接受。这是所有权语义的核心行为。
- **T-2 移动赋值基本运行时测试**：接受。不能只依赖旁路测试覆盖核心语义。
- **T-3 `reset(R)` 值语义测试**：接受。应覆盖普通值资源场景。
- **T-4 `operator*` / `operator->` 运行时测试**：接受。编译探测不足以证明行为正确。
- **T-5 编译探测使用 `requires` 而非 C++17 SFINAE**：接受。原因不是代码风格，而是测试本身也应使用与项目目标一致的表达方式来验证约束。
- **T-7 CTAD 测试**：接受，且优先级应上调。若 `I-3` 最终移除，则 CTAD 会成为用户主要的简洁构造方式，其正确性直接影响标准一致性与使用体验。
- **T-8 ADL `swap` 自由函数测试**：接受。标准接口的使用方式应被直接验证。
- **T-9 不应硬编码 copy 次数**：接受。测试应绑定标准语义，而非绑定具体实现细节。
- **T-10 `reset(R)` 异常安全测试**：接受。高优先级。
- **T-11 不可拷贝语义测试**：接受。属于关键静态契约。
- **T-12 checked 构造后重新激活测试**：接受。该状态转换行为应被覆盖。
- **T-13 通过 `forge::forge` 与 `<memory>` 验证注入机制**：接受。这一点对本项目尤其重要，不应只测实现头直引路径。
### 可接受，且优先级较低
- **T-6 使用 `_v` 而非 `::value`**：接受，但优先级较低。它更多是代码一致性整理。
